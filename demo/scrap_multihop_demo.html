SCRAP — Accountability across a federated operator mesh

This is the Satlyt marketplace at scale: one customer task crossing Operator A → B → C, each independently paid and accountable. SCRAP carries it onion-wrapped, so each hop sees only the next, and assembles end-to-end proof from the parties' public keys. The mechanism is countersigned receipts: every hop signs custody when it receives the bundle, and each hop's execution is vouched by the next hop's signature rather than its own. A governance body like Kenya's NEODC can replay the whole chain afterward.

1The route source-routed, cross-operator

The customer's operator computes the path and pre-signs an attenuated capability token for each hop (relay-only inland, execute at the end) — each a strict subset of the root grant.

2Onion-wrapped forward path spec §5.5

The task bundle is wrapped in layers (innermost = final hop). Each hop peels one layer with its link key, sees only its instructions + the next hop, and forwards the inner blob. No satellite holds the full route.

3Countersigned receipt chain the backward flow

Two signed families flow back. Custody: each hop signs "I received this bundle from ⟨prev⟩" the moment it arrives, before it acts. Output-ack: each hop signs "I received ⟨prev⟩'s output ⟨hash⟩", so a hop's execution status is attested by the next hop in the chain. 4Backward settlement cascade spec §5.4

Payment routes on operator Lightning channels, not ISL. The payee reveals preimage R (where SHA256(R)=H); it cascades backward node-3 → node-2 → node-1 → customer, settling each channel. Decreasing HTLC timeouts force the payee-claims-first ordering. No escrow, no arbiter.
